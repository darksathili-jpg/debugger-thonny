Séance 1

Comprendre les bugs

1 heure
📚 Cours — Trois familles d'erreurs

Quand un programme ne fonctionne pas, il existe exactement trois situations possibles. Les distinguer, c'est déjà à moitié résoudre le problème.

Situation 1 — Python refuse même de démarrer

Tu cliques sur « Exécuter » et Python s'arrête avant même d'avoir fait quoi que ce soit. Il affiche un message du type SyntaxError. Pourquoi ? Parce que ton code ressemble à une phrase avec une faute de grammaire tellement grave que Python ne peut pas l'interpréter.

Exemples courants : une parenthèse oubliée, un guillemet non fermé, une indentation incorrecte.

Erreur de syntaxe — le programme ne démarre pas.

Situation 2 — Python démarre, puis s'effondre

Cette fois le programme se lance, mais à un moment précis il bloque avec un message d'erreur. Python a rencontré quelque chose d'impossible à faire, par exemple diviser par zéro ou accéder à un indice qui n'existe pas dans une liste.

Erreur d'exécution — le programme démarre mais s'arrête en cours de route.

Situation 3 — Le programme tourne mais le résultat est faux

La situation la plus sournoise. Le programme s'exécute sans aucun message d'erreur, mais la réponse affichée est incorrecte. Il y a une erreur dans la logique : une condition mal formulée, une variable mal initialisée, une boucle qui tourne une fois de trop…

Erreur logique — le programme termine, mais le résultat est faux.

📚 Cours — Lire un message d'erreur Python

Un message d'erreur Python n'est pas un insulte : c'est une piste de correction. Il suit toujours la même structure. Regarde cet exemple : 

Traceback (most recent call last):
  File "jeu.py", line 7, in <module>
    resultat = score / nb_parties
ZeroDivisionError: division by zero

Décode-le ligne par ligne :

  1. Traceback — c'est la « trace » du chemin parcouru avant le crash.
  2. File "jeu.py", line 7 — le problème est à la ligne 7 du fichier jeu.py.
  3. resultat = score / nb_parties — voilà l'instruction exacte qui a causé l'erreur.
  4. ZeroDivisionError: division by zero — la cause : on a divisé par zéro.
💡 Réflexe à adopter

Lis toujours la dernière ligne en premier : elle te dit quoi s'est passé. Puis remonte pour savoir où.

Séance 2

Prendre en main Thonny

1 heure

Tu sais maintenant reconnaître un bug. Il est temps d'apprendre à le traquer avec un outil conçu exactement pour ça.

📚 Cours — L'interface de débogage de Thonny

Thonny est un environnement Python pensé pour l'apprentissage. Son point fort : un mode débogage qui te permet d'exécuter ton programme une instruction à la fois, comme un film qu'on peut mettre en pause à chaque image.

Les éléments essentiels de l'interface

Zone Rôle Raccourci
Bouton Débogage Lance le programme en mode pas à pas Ctrl + F5
Step over Exécute l'instruction en cours sans entrer dans les fonctions F6
Step into Entre à l'intérieur d'une fonction appelée F7
Panneau Variables Affiche la valeur de toutes les variables à l'instant T Menu View
Panneau Stack Affiche la pile d'appels de fonctions en cours Menu View
Surlignage bleu Indique la prochaine instruction qui sera exécutée Automatique
💡 Conseil pratique

Active les panneaux « Variables » et « Stack » dès le début via le menu View. Sans eux, le débogage ressemble à conduire les yeux bandés.

📚 Cours — Deux modes de débogage

Thonny propose deux variantes du mode débogage, selon la situation :

Mode Nicer (recommandé pour débuter)

Thonny décompose chaque expression en sous-expressions et les met en évidence une à une. C'est comme si on décortiquait le calcul 2 + 3 * 4 en montrant d'abord 3 * 4 = 12, puis 2 + 12 = 14. Très visuel, légèrement plus lent.

Mode Faster (pour programmes plus longs)

Thonny avance instruction par instruction, sans décomposer les expressions. Plus rapide, mais moins détaillé. À utiliser quand tu as déjà une idée de la zone à inspecter.

Séance 3

Pas-à-pas et état mémoire

1 heure

Un programme n'est pas une boîte noire qui produit un résultat magique. C'est une suite d'états : à chaque instruction, la mémoire change. Savoir lire ces états, c'est comprendre ce que le programme fait vraiment, pas ce qu'on croit qu'il fait.

📚 Cours — Tracer l'exécution à la main

Avant même d'ouvrir Thonny, il est très utile de savoir prévoir le comportement d'un programme sur papier. Cette technique s'appelle la trace d'exécution.

On crée un tableau avec une colonne par variable, et une ligne par instruction importante. À chaque ligne, on note la nouvelle valeur de la variable qui vient d'être modifiée. Si une variable ne change pas, on laisse sa cellule vide (ou on recopie la valeur précédente).

Exemple — Calcul de la somme des nombres de 1 à 5

somme = 0
for i in range(1, 6):
    somme = somme + i
print(somme)

✎ Exercice 5 — Repérer le moment où ça diverge

La fonction ci-dessous est censée calculer le maximum d'une liste. Elle contient un bug logique : à un moment précis de son exécution, la valeur de maximum n'est plus celle attendue.

🔎 Indice 2

✅ Correction — Exercice 5 
def trouver_maximum(liste):
    maximum = liste[0]   # on part du premier élément
    for valeur in liste:
        if valeur > maximum:
            maximum = valeur
    return maximum

def test_trouver_maximum():
    assert trouver_maximum([3, 7, 2, 9, 1]) == 9
    assert trouver_maximum([-5, -2, -8])    == -2
    assert trouver_maximum([0, 0, 1])       == 1
    print("Les tests sont validés.")

Séance 4

Les points d'arrêt

1 heure

Avancer instruction par instruction sur un programme de 200 lignes, c'est épuisant. Il doit exister une façon plus intelligente de travailler. Et il y en a une.

📚 Cours — Les points d'arrêt (breakpoints)

Imagine que tu saches que ton bug se situe quelque part autour de la ligne 80. Devoir appuyer 79 fois sur F6 pour y arriver est absurde. Un point d'arrêt résout ce problème : tu marques la ligne 80, tu lances le débogueur, et le programme s'arrête automatiquement juste avant d'exécuter cette ligne. Tu inspectes alors les variables directement dans leur état à cet instant précis.

Placer un point d'arrêt dans Thonny

  1. Clique dans la marge grise à gauche du numéro de ligne — un point rouge apparaît.
  2. Lance le débogueur normalement (Ctrl + F5).
  3. Le programme s'exécute à vitesse normale jusqu'au point d'arrêt, puis se met en pause.
  4. Inspecte le panneau Variables. Continue avec F6 si nécessaire.
  5. Pour supprimer un point d'arrêt, clique à nouveau sur le point rouge.
💡 Stratégie de placement

Place ton premier point d'arrêt au milieu du code, pas au début. Si les variables sont correctes à la moitié, le bug est dans la seconde moitié. Sinon, il est dans la première. Tu as divisé le problème en deux — c'est une recherche dichotomique appliquée au débogage !

✎ Exercice 6 — Identifier la ligne d'un breakpoint optimal

Le programme suivant est censé afficher la liste des multiples de 3 inférieurs à 20. Il affiche en réalité [3, 6, 9, 12, 15, 18, 21] — un élément de trop. Réponds aux questions sans modifier le code. 

multiples = []
i = 3
while i <= 20:
    multiples.append(i)
    i = i + 3
print(multiples)
  1. À quelle ligne placerais-tu un point d'arrêt pour surveiller l'état de multiples à chaque tour de boucle ?
  2. Quelle est la valeur de i au moment où l'élément 21 est ajouté à la liste ?
  3. Quelle correction faut-il apporter à la condition du while ?
🔎 Indice 1

Place le breakpoint à la ligne 4 (multiples.append(i)). Chaque fois que le programme s'y arrête, observe la valeur de i.

🔎 Indice 2

Quand i = 21, la condition 21 <= 20 est fausse… mais non : le programme entre quand même dans la boucle parce que la vérification se fait avant d'incrémenter i. La condition correcte est i < 20.

⏱Correction disponible dans 2:00
✅ Correction — Exercice 6

La condition while i <= 20 laisse entrer la valeur 21 car au dernier tour, i vaut 18, on ajoute 18, puis i passe à 21 — mais 21 ≤ 20 est faux.

Attends — relis la trace : après avoir ajouté 18, i devient 21. La condition 21 <= 20 est fausse, donc la boucle s'arrête. L'erreur est différente : la condition est i <= 20, donc 21 n'est pas ajouté. La liste affichée [3,6,9,12,15,18,21] indiquerait que la condition devrait être i < 20 (strict) pour exclure 21.

Correction :

multiples = []
i = 3
while i < 20:   # < strict pour exclure 21
    multiples.append(i)
    i = i + 3
print(multiples)  # [3, 6, 9, 12, 15, 18]

def compter_voyelles(mot):

Séance 5

Déboguer les boucles et les conditions

1 heure

Tu maîtrises maintenant les outils. Il est temps d'affronter les bugs les plus fréquents en programmation Python : ceux qui se cachent dans les boucles et les conditions.

📚 Cours — Les pièges classiques

Piège 1 — La boucle infinie

Une boucle while tourne indéfiniment si la condition de sortie n'est jamais atteinte. Cause habituelle : la variable de contrôle n'est jamais modifiée à l'intérieur de la boucle. 

# Boucle infinie — ne jamais exécuter !
compteur = 0
while compteur < 10:
    print(compteur)
    # oubli : compteur = compteur + 1

Piège 2 — La variable non réinitialisée

Quand une même variable sert d'accumulateur pour plusieurs calculs successifs, il faut la remettre à zéro entre chaque calcul. Sinon, les résultats se « contaminent » mutuellement.

Piège 3 — L'erreur « off-by-one »

Une boucle itère une fois de trop ou une fois de moins. Cause classique : confusion entre < et <=, ou entre range(n) (0 à n-1) et range(1, n+1) (1 à n).

Piège 4 — La condition toujours vraie ou toujours fausse

Un if dont la condition est logiquement incohérente (par exemple tester un entier avec == alors que la valeur est un flottant) ne s'exécutera jamais, ou s'exécutera toujours.

✎ Exercice 8 — Boucle et variable mal réinitialisée

La fonction somme_par_groupe doit calculer la somme des entiers dans chaque sous-liste. Elle renvoie des valeurs incorrectes à partir du deuxième groupe.

🔎 Indice 1

Trace l'exécution à la main avec groupes = [[1,2,3],[4,5]]. Quelle est la valeur de total au début du deuxième groupe ?

🔎 Indice 2

Il faut déplacer total = 0 à l'intérieur de la boucle externe, juste avant la boucle interne, pour repartir de zéro à chaque nouveau groupe.

Lance les tests dans l'éditeur ci-dessus pour débloquer la correction.
✅ Correction — Exercice 8 
def somme_par_groupe(groupes):
    resultats = []
    for groupe in groupes:
        total = 0   # réinitialisation à chaque groupe
        for valeur in groupe:
            total = total + valeur
        resultats.append(total)
    return resultats

def test_somme_par_groupe():
    groupes = [[1, 2, 3], [4, 5], [10]]
    assert somme_par_groupe(groupes) == [6, 9, 10]
    assert somme_par_groupe([[0], [0, 0]]) == [0, 0]
    assert somme_par_groupe([[7, 3], [2]]) == [10, 2]
    print("Les tests sont validés.")

test_somme_par_groupe()

Séance 6

Déboguer les fonctions

1 heure

Tu sais traquer un bug dans un script linéaire. Mais dès qu'une fonction est appelée, quelque chose change : le programme « entre » dans un nouveau contexte, avec ses propres variables. C'est là qu'intervient la pile d'appels.

📚 Cours — La pile d'appels (call stack)

Quand Python appelle une fonction, il met le programme principal « en pause » et crée un nouvel espace de variables pour la fonction. Quand la fonction termine, Python revient à l'endroit exact où il s'était arrêté. C'est comme empiler des cartes : on pose une nouvelle carte à chaque appel, on l'enlève quand la fonction renvoie son résultat.

Le panneau Stack de Thonny affiche cet empilement en temps réel. Chaque ligne représente un cadre d'appel. Le cadre du haut est la fonction en cours d'exécution. Le cadre du bas est le programme principal.

Variables locales vs variables globales

Une variable définie à l'intérieur d'une fonction n'existe que le temps de l'exécution de cette fonction. Elle disparaît dès que la fonction se termine. C'est une variable locale. Les variables définies hors de toute fonction sont des variables globales.

💡 Erreur classique

Modifier une variable globale à l'intérieur d'une fonction sans déclarer global ne fonctionne pas comme attendu en Python. La bonne pratique : passer les données en paramètres et les récupérer via return.

📚 Cours — Step into vs Step over

Quand le débogueur arrive sur un appel de fonction, tu as le choix :

  • F6 — Step over : Thonny exécute la fonction entière d'un coup et passe à la ligne suivante. Utile quand tu fais confiance à la fonction.
  • F7 — Step into : Thonny entre à l'intérieur de la fonction, ligne par ligne. Le panneau Stack change pour montrer le cadre de la fonction. Utilise-le quand tu suspectes que le bug est dans la fonction.
✎ Exercice 10 — Une fonction qui marche seule mais pas dans le programme

Ce type de situation est très courant : on teste une fonction isolément et elle semble fonctionner, mais quand on l'intègre dans un programme plus grand, le résultat est faux. Trouve pourquoi.

🔎 Indice 1

La fonction doubler(v) renvoie bien le double de v. Mais est-ce que ce résultat est utilisé quelque part ?

🔎 Indice 2

Appeler doubler(v) sans récupérer le résultat, c'est comme commander un café et laisser le barista le poser sur le comptoir sans jamais le prendre. Il faut écrire resultat = doubler(v) puis resultats.append(resultat).

Lance les tests dans l'éditeur ci-dessus pour débloquer la correction.

    assert appliquer_sur_liste([0, 5])    == [0, 10]
    assert appliquer_sur_liste([7])       == [14]
    print("Les tests sont validés.")

✎ Exercice 11 — Variable locale vs variable globale

Ce programme est censé augmenter un score global. Il ne fonctionne pas. Identifie pourquoi, puis propose deux solutions : l'une avec global, l'autre sans (avec return).

Lance les tests dans l'éditeur ci-dessus pour débloquer la correction.
✅ Correction — Exercice 11

Pourquoi ça plante ?

En Python, quand tu écris score = score + points dans une fonction, Python voit que score est assigné dans la fonction et le considère automatiquement comme une variable locale. Mais il essaie de lire score avant de l'assigner (pour faire l'addition), ce qui produit une UnboundLocalError.

Solution 1 — avec global (déconseillée en général) :

score = 0

def ajouter_points(points):
    global score       # déclare qu'on utilise la variable globale
    score = score + points

ajouter_points(10)
ajouter_points(5)
print(score)  # 15

Solution 2 — avec return (recommandée) :

def ajouter_points(score_actuel, points):

📚 Cours — La méthode en 7 étapes
  1. Décrire le résultat attendu. Avant même de chercher le bug, formule clairement ce que le programme devrait produire. « La fonction doit renvoyer 42 pour l'entrée 7. »
  2. Reproduire le bug de façon fiable. Identifie quelle entrée déclenche le problème. Un bug qu'on ne peut pas reproduire ne peut pas être corrigé.
  3. Lire le message d'erreur ou observer l'anomalie. Lis la dernière ligne du message en premier : c'est le type d'erreur. Remonte pour trouver la ligne concernée.
  4. Placer un point d'arrêt au bon endroit. Commence au milieu de la zone suspecte. Si les variables sont correctes à ce point, le bug est plus loin.
  5. Suivre les variables pas à pas. Avance avec F6 ou F7 et observe chaque changement de valeur. Cherche le moment exact où une valeur diverge de ce qui était attendu.
  6. Corriger une seule chose à la fois. Ne modifie qu'une seule ligne avant de re-tester. Si tu corriges plusieurs choses à la fois, tu ne sais plus ce qui a résolu le problème.
  7. Rejouer le test complet. Après correction, relance tous les tests — y compris ceux qui passaient déjà. Une correction peut en casser une autre.

def calculer_moyenne(scores):
    total = 0              # correction 1
    for s in scores:
        total = total + s
    return total / len(scores)

def trouver_meilleur(scores):
    meilleur = scores[0]
    for s in scores:
        if s > meilleur:    # correction 2
            meilleur = s
    return meilleur

def compter_au_dessus(scores, seuil):
    compteur = 0
    for s in scores:
        if s > seuil:
            compteur = compteur + 1
    return compteur          # correction 3

def afficher_bilan(scores):
    dernier = scores[-1]      # correction 4
    print("Dernier score :", dernier)
    print("Moyenne       :", calculer_moyenne(scores))
    print("Meilleur      :", trouver_meilleur(scores))
    print("Au-dessus de 10 :", compter_au_dessus(scores, 10))
